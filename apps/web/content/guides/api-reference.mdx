# API Reference

Capsa renders a full interactive API reference from an OpenAPI document using
[Scalar](https://github.com/scalar/scalar). It lives at **/docs/api** and is
lazy-loaded, so it stays out of the main bundle until visited.

## Add your spec

1. Drop your OpenAPI document at `apps/web/public/openapi/v1.yaml`
   (replacing the example spec).
2. That's it — visit `/docs/api`.

The **API Reference** tab in the top bar links there; it's defined in
`src/navigation.ts`:

```ts
{ tab: 'API Reference', icon: 'code', href: '/docs/api' }
```

## Versions

Register each version in `src/apiVersions.ts`:

```ts title="src/apiVersions.ts"
export const apiVersions: ApiVersion[] = [
  { label: 'v1.0', url: '/openapi/v1.yaml' },
  // { label: 'v2.0', url: '/openapi/v2.yaml' },
];
```

With a single entry there's no switcher. Add a second and a version **dropdown**
appears on the reference page automatically.

## "Try it"

Scalar's built-in API client is enabled — readers can execute requests with a
key they paste (Scalar stores it locally). To point it at a live sandbox, add a
`servers` entry to your spec and enable CORS for the docs origin on that server.

## Keep the spec in sync

If your spec is generated by an API repo, you don't have to copy it by hand. The
included `scripts/sync-openapi.mjs` copies a spec into `public/openapi/v1.yaml`
when `OPENAPI_SOURCE` is set — wire it into your build or a CI step:

```bash
OPENAPI_SOURCE=../api/openapi.yaml pnpm dev:web
```

A common pattern: the API repo commits its spec to this repo (or triggers a
rebuild) on change, so the reference is always current.

## Inline references in prose

Use the `<ApiMethod>` component in any guide to link to an endpoint:

<ApiMethod method="POST" path="/v1/widgets" desc="Create a widget" />
